## 2026-03-29 - [Fail-Secure Configuration Checks]
**Vulnerability:** Hardcoded credentials (`admin:s3cret`) were present in documentation examples and actual example code (`example/main.go`). A naive fix of just using `user == cfg.Monitor.Username && pass == cfg.Monitor.Password` without an explicit existence check allows an authentication bypass if the configuration is omitted (empty username and password match empty string inputs).
**Learning:** Examples and documentations are often copied verbatim into production by developers. Hardcoded secrets in examples present a realistic vulnerability if not modified. Replacing them with configuration-driven checks introduces a secondary risk of authentication bypass (fail-open) if the environment variables are not supplied.
**Prevention:** When replacing hardcoded credentials with config-driven variables, always implement a fail-secure check (e.g., `cfg.Username == "" || cfg.Password == ""`) to explicitly reject authentication attempts when the system is unconfigured, preventing "empty string" bypasses. Treat documentation and example code with the same security rigor as production code.
